# AIConfigurator：多框架 LLM 推理的闪电般快速配置优化

> 原文：[arXiv:2601.06288](https://arxiv.org/abs/2601.06288)
>
> 作者：Tianhao Xu, Yiming Liu, Xianglong Lu 等（NVIDIA）
>
> 翻译日期：2026-04-04

## 摘要

在生产系统中优化大语言模型（LLM）推理变得越来越困难，原因包括动态工作负载、严格的延迟/吞吐目标，以及快速扩展的配置空间。这种复杂性不仅涵盖分布式并行策略（张量/流水线/专家并行），还涉及复杂的框架特定运行时参数，例如 CUDA graphs 的启用、可用 KV-cache 内存比例和最大 token 容量等，这些参数对性能有巨大影响。现代推理框架（如 TRT-LLM、vLLM、SGLang）各自采用不同的内核和执行策略，使得手动调优既与框架紧密绑定，又在计算上代价高昂。

我们提出了 **AIConfigurator**，一个统一的性能建模系统，能够在**不需要 GPU 实测**的情况下实现快速、框架无关的推理配置搜索。AIConfigurator 结合了：(1) 将推理分解为可分析建模的基本操作原语（GEMM、注意力、通信和内存操作）的方法论，同时捕获框架特定的调度动态；(2) 跨多种硬件平台和主流开源模型（GPT-OSS、Qwen、DeepSeek、LLaMA、Mistral）的校准内核级性能数据库；(3) 自动为目标后端解析最优启动参数的抽象层，无缝集成到生产级编排系统。

在生产 LLM 推理工作负载上的评估表明，AIConfigurator 识别出的优越推理配置可将 Dense 模型（如 Qwen3-32B）性能提升高达 **40%**，MoE 架构（如 DeepSeek-V3）性能提升高达 **50%**，同时平均在 **30 秒**内完成搜索——实现从集群拓扑到引擎特定标志的广阔设计空间的快速探索。

## 1 引言

大语言模型的快速演进对计算基础设施提出了前所未有的要求。随着最先进模型的参数量从数亿扩展到数千亿，推理部署的效率已成为经济可行性的关键决定因素。然而，优化这些部署充满了复杂性。服务提供商必须在配置参数的组合爆炸中导航——从张量并行、流水线并行和专家并行策略到批大小和量化的细粒度设置。传统的性能调优方法依赖于手动基准测试和穷举测试，这越来越不可行。随着现代 GPU 成本的上升，穷举测试在经济上变得不可接受。

**解聚推理**（disaggregated serving）的出现——将 prefill 和 decode 阶段分配到不同的计算资源上——进一步复杂化了这一局面。虽然解聚推理承诺在严格延迟约束下优化"有效吞吐量"（Goodput），但它并非普遍优越的解决方案；收益在很大程度上取决于模型架构、硬件拓扑和网络带宽之间的交互。从业者面临一个困难的权衡：对于特定工作负载，解聚的调度灵活性是否超过了通信开销？此外，配置此类系统以满足严格的服务等级协议（SLA）——特别是首 Token 时间（TTFT）和每输出 Token 时间（TPOT）——创建了一个可能超过 **10,000 种排列组合**的设计空间。

除了架构决策之外，推理引擎本身的复杂性进一步加剧了问题。现代框架如 TensorRT-LLM、vLLM 和 SGLang 暴露了大量可调的运行时标志，例如 CUDA graphs 的启用、可用 KV-cache 内存比例和最大 token 容量等。最大化特定工作负载吞吐量的配置在推理环境演变时往往变得脆弱或低效。

虽然黑盒优化方法（如 Vizier）或自动化推理框架（如 Morphling）可以加速寻找最优配置，但它们仍然需要大量 GPU 小时才能对每个特定场景收敛到解决方案。

### 我们的贡献

- 设计了一个能够在**数秒内**以高精度导航复杂推理配置空间的系统
- 通过与主流推理框架（vLLM、SGLang、TRTLLM 和 NVIDIA Dynamo）的无缝集成证明了系统的有效性
- 通过与真实硅片数据的基准对比进行了全面评估

## 2 背景

### 2.1 LLM 推理优化

LLM 推理优化涉及三个相互依存的支柱：

1. **高级调度**——包括连续批处理（continuous batching）、PagedAttention、分块预填充（chunked prefills）和解聚推理，通过解决 prefill（计算密集）和 decode（内存密集）阶段的不同计算特征来最大化硬件利用率
2. **模型并行**——通过张量并行（TP）、流水线并行（PP）和专家并行（EP，用于 MoE 架构）将大模型分布到多个 GPU
3. **配置调优**——导航由此产生的组合空间；虽然模拟器如 Vidur 和 APEX 能实现快速探索，但它们依赖于理论 roofline 模型，往往与生产性能存在偏差

### 2.2 聚合推理 vs 解聚推理

**解聚推理**将 prefill 和 decode 分配到不同的 GPU 池，实现独立扩展，但引入了 KV-cache 传输开销。Splitwise 的研究表明，对于短上下文，这种开销可能抵消收益。最优架构取决于：
- 工作负载组合（prefill 密集 vs decode 密集）
- 互连带宽
- 集群规模

对于较小的部署，带分块预填充的聚合推理往往优于解聚方案。

## 3 动机

传统启发式规则如"节点内用 TP，节点间用 PP"无法捕捉计算和网络带宽之间的非线性交互。研究表明自动搜索可以比手动调优提高 **2 倍以上**的成本效率。

**框架异构性：** 生产推理跨越多个框架：
- **vLLM**：PagedAttention，Python 调度
- **SGLang**：RadixAttention，Triton 内核
- **TensorRT-LLM**：静态图优化，自定义内核
- **NVIDIA Dynamo**：后端无关的编排

每个框架都展现出独特的性能悬崖，受大量框架特定运行时标志控制。

**配置调优差距：** 当前方法（夜间基准测试和策划的配方）是静态查找表，不足以应对动态生产环境。

## 4 设计与实现

AIConfigurator 采用有原则的、数据驱动的方法来导航庞大的 LLM 推理配置空间。系统将推理分解为基本操作，收集这些原语的真实硬件测量数据，并通过定义良好的性能模型组合端到端性能估计。

### 4.1 AIConfigurator 工作流程

工作流程包括五个步骤：

1. **PerfDatabase（性能数据库）**：离线数据收集过程，为指定硬件平台上常用 LLM 算子构建综合性能数据库。每个支持的推理框架独立处理
2. **TaskRunner（任务运行器）**：根据用户提供的工作负载描述符（环境设置、SLA、请求特征），构建所有有效候选推理配置的搜索空间
3. **InferenceSession（推理会话）**：遍历所有候选配置，通过结合迭代级建模和从 PerfDatabase 查询的算子级性能数据来估计关键性能指标
4. **Pareto Analyzer（帕累托分析器）**：根据性能预测筛选和排名所有有效配置，输出带有综合性能预测的最优配置
5. **Generator（生成器）**：将推荐配置直接转换为与版本兼容的推理引擎启动文件（TensorRT-LLM、vLLM 或 SGLang），自动设置最优标志如 `--enable_cuda_graph`、`--kv_cache_free_gpu_mem_fraction` 和 `--enable_chunked_context`

### 4.2 性能建模

AIConfigurator 从稳态并发下到达服务器的平均请求的角度推导系统吞吐量和生成速度：

```
生成速度 = 1000 / TPOT

系统吞吐量 = 1000 / (TTFT + (OSL-1) × TPOT) × BatchSize × OSL / GPU总数
```

AIConfigurator 明确建模三种不同的推理模式：

#### 4.2.1 静态模式
工作负载以固定批大小严格顺序处理。TTFT 等于 prefill 阶段的延迟，TPOT 通过对后续解码步骤的延迟取平均来近似。使用步幅优化减少估计生成延迟的计算开销。

#### 4.2.2 聚合模式（连续批处理）
能够在单个推理迭代中混合来自不同请求的 prefill 和 decode 步骤。分为两个阶段：
- **混合阶段**：prefill 和 decode 请求并发运行的稳态操作
- **纯生成阶段**：prefill 队列排空后的纯解码模式

使用经验校正因子估计 TTFT，使用两个阶段延迟的加权平均估计 TPOT。

#### 4.2.3 解聚模式
使用两个独立的工作池，分别专用于 prefill 和 decode 阶段。建模分两步：
1. 独立扫描 prefill 和 decode 配置空间，将每个候选作为隔离的静态实例估计基础延迟
2. 通过速率匹配过程构建有效的组合服务器（表示为 (x)P(y)D），最大化每 GPU 有效吞吐量

### 4.3 迭代级建模

由于现代 LLM 由重复的 Transformer 层组成，任何推理迭代步骤都可以建模为固定序列的算子重复执行若干次。例如，典型的 MoE 模型推理步骤本质上归结为运行 4 种类型的算子：embedding、GEMM、Attention（prefill 或 decode）和 MoE。

引入现代并行策略不会改变这一基本属性，只是在迭代执行路径的固定位置插入少量定义明确的通信算子，并通过跨多个计算设备分片输入来缩减计算算子的规模。

### 4.4 算子数据库

AIConfigurator 通过在实际 GPU 硬件上进行离线性能分析来构建性能数据库，当前支持 TensorRT-LLM、vLLM 和 SGLang。

**数据库覆盖范围：**
- **GEMM 操作**：按维度 (M,N,K) 和量化（FP16、FP8、INT8、INT4）参数化
- **注意力操作**：计算密集的上下文注意力和内存密集的生成注意力，支持 MHA、GQA 和 MLA
- **通信原语**：AllReduce、AllGather、AllToAll 和点对点传输
- **MoE 操作**：dispatch/combine 模式
- **硬件规格**：内存带宽、计算吞吐量、互连带宽

**数据收集策略：**
- **穷举分析**：用框架原生工具扫描参数（每个平台-框架组合约 30 GPU 小时）
- **插值**：使用已分析的数据点估计中间配置的延迟
- **光速估计**：通过 roofline 模型为未分析的算子提供分析边界

#### 4.4.1 MoE 的幂律校正

MoE 算子性能严重依赖 token 分布。先前研究表明，在推理期间某些专家接收不成比例的更多 token——来自 Qwen3-235B 的观察表明约 **70% 的计算由仅 20% 的活跃专家**处理。

为了解释这种不平衡，AIConfigurator 实现了一个受控的 token 分配过程：
1. **采样专家负载权重**：从幂律分布中采样，参数 α 控制不平衡程度（α≈0 为近似均匀，α≈1.2 为重尾分布）
2. **构建合成路由分配**：绕过学习到的门控网络，直接注入确定性的分配矩阵，消除路由器的随机方差

## 5 评估

### 5.1 聚合推理保真度

**实验设置：** 单个 NVIDIA H100 SXM 节点（8 GPU，每个 80GB HBM3，通过 NVSwitch 连接）。评估两个模型：
- Qwen3-32B（320 亿参数，Dense，FP8）
- Qwen3-235B（2350 亿参数，MoE，128 个专家，FP8）

配置空间：ISL 128-4096，OSL 128-512，并发 4-128，TP 1/2/4/8，EP 1/2/4/8。共 960 个 TensorRT-LLM 配置 + 128 个 vLLM 配置。

**结果：**
- **TPOT 预测**总体 MAPE 为 **7.8%**：
  - Qwen3-32B-TRTLLM：8.2%，r=0.96
  - Qwen3-235B-MoE-TRTLLM：6.8%，r=0.98
  - Qwen3-32B-VLLM：11.9%，r=0.99
- **TTFT 预测**：
  - Qwen3-32B-TRTLLM：22.1% MAPE，r=0.89
  - Qwen3-235B-MoE-TRTLLM：18.3% MAPE，r=0.66
  - Qwen3-32B-VLLM：16.9% MAPE，r=0.95

MoE 模型实现了最低的 TPOT 误差，验证了幂律工作负载建模的有效性。

### 5.2 解聚推理保真度

**实验设置：** 两个计算节点（各 8 个 NVIDIA Hopper GPU，通过 NVLink 互连）。评估完整 DeepSeek V3（6710 亿参数，MoE）。

**结果：**
- 总体 MAPE：系统吞吐量 25.49%，生成速度 14.94%
- 在交互速度区域（25-50 tokens/s/user，大多数用户的舒适阅读速度）内，保真度显著提高：吞吐量 MAPE 降至 **13.19%**，生成速度 MAPE 降至 **3.35%**

### 5.3 搜索效率

| 模型 | 配置数量 | AIConfigurator 时间 | GPU 基准测试时间 | 加速比 |
|------|---------|-------------------|----------------|--------|
| Qwen3-32B | 128 | ~0.2s | ~8.5 GPU 小时 | ~153,000× |
| Qwen3-235B | 506 | 0.8s | 99.5 GPU 小时 | **427,000×** |

每个配置的模拟时间中位数保持恒定在约 **1.5ms**，与模型大小无关。

### 5.4 案例研究：寻找最优配置

**场景：** Qwen3-32B-FP8 在 8 个 H200 GPU 上，SLA 要求 TTFT ≤ 1200ms，生成速度 ≥ 60 tokens/s/user。

| 配置 | 吞吐量 (tokens/s/GPU) | 生成速度 (tokens/s/user) |
|------|----------------------|------------------------|
| 聚合最优（TP2，batch=8） | 321.5 | 95.9 |
| 解聚最优（4P-TP1 + 2D-TP2） | **648.3** | 72.3 |

解聚配置实现了 **101.6%** 的吞吐量提升，同时满足所有 SLA 约束。

## 6 相关工作

- **LLM 推理系统**：从静态批处理演进到动态调度（Orca 连续批处理、vLLM PagedAttention、SGLang RadixAttention、TensorRT-LLM 优化内核）
- **性能模拟**：Vidur 和 APEX 通过离散事件模拟实现快速配置探索，但依赖理论 roofline 模型。AIConfigurator 通过数据驱动基础和框架特定开销的实测来区别于它们

## 7 结论

AIConfigurator 是一个数据驱动的工具包，用于优化跨 TensorRT-LLM、vLLM 和 SGLang 的 LLM 推理配置。通过将推理分解为基本操作并在目标硬件上测量其延迟，AIConfigurator 实现了高保真的性能预测，捕获了分析模拟器遗漏的框架特定开销。该工具包在 CPU 上数秒内评估数千种配置，消除了昂贵的 GPU 基准测试活动。

评估表明，TPOT 预测精度达到 6-12% MAPE，跨 Dense 和 MoE 架构表现强劲。生产案例研究表明，通过自动化解聚配置发现可实现 **2 倍吞吐量提升**。
